Pulse 架构 — 逐跳、head 作为端点、durable 作为连接能力

把长讨论收敛成一份可 review 的设计。重点在纠正一个我之前的地基错误,并把 durable / E2E / relay 的关系彻底摆正。
UPDATE — pulse 层的 durable 已经做完了(TS 92 + Swift 47 测试全绿) 下面第 12 节"待你拍的决定"已全部拍定并落地。最终形态与本文正文有三处偏差,以此处为准:
  • 最小 durable 集 = 只 delete_session(mode/model 先靠 session_list 兜)。
  • store/unstore 归 pulse 库,且核只发 {seq, bytes} —— 零目标、零路由。durable = "我的 outbox 这条要持久化",不是"存给第三方"。这是守住 kraki-free 的关键。
  • fail-fast 不靠 server_error(正文第 8 节 approve 走查 + #3 写错了):tentacle 离线时 head 在线,arm→head 成功,不存在发送失败。机制 = arm 的 pulse 等不到 acked 超时 → 应用回滚。head 保持哑。
  • durable 是把已有能力从 head refactor 进 pulse,不是发明新东西:head 的 pending_messages + tentacle 的 processPendingMessages 本就是它的手写前身。判定从"接收端取时筛"前移到"发送端 send 时标 flag"。

1 · 三句话结论

① pulse 是逐跳的,不是端到端的 —— 它跑在每条真实 WS 上(arm↔head、tentacle↔head), head 是同时参与两条、且总在线的 pulse 端点。
② durable 是"连接端点的一个能力" —— 握手时协商,head 声明支持、arm/tentacle 声明不支持;发消息时若对端支持,才能给这条打 durable。
③ E2E 与 pulse 正交 —— 加密是 kraki 层的事,pulse 运的 payload 恰好是密文,但 pulse 不知情;head 读 pulse 传输头,读不了 kraki 内容。

一句话 relay 在 pulse 眼里不是"中转站"这种特殊拓扑,它就是连接的另一端,一个碰巧声明了 durable=true 的普通 pulse 端点。这就是整套设计不破 pulse 中立哲学的原因。

2 · 我之前错在哪(得认)

我前面几轮的集成,是把 pulse frame 塞进 E2E 密文的 payload 里,让它"端到端"从 arm 穿过 head 到 tentacle, head 当哑管道透传。这等于我假装 arm 和 tentacle 之间有一条逻辑 pulse 连接。

地基错误 arm 和 tentacle 之间根本没有 WS。它俩从不直接连。物理上 WS 的两端只有 arm↔head 和 tentacle↔head。 pulse 绑在 WS 连接上,所以 pulse 端点只能长在 head 的两条 WS 上 —— head 本来就必须是 pulse 端点。 我把 pulse 架在了一条不存在的连接上,由此引出了后面所有别扭:durable 没处放、head pending 和 pulse 打架、发送方离线没人扛。

下面全部是把 pulse 挪回正确连接层之后的设计。

3 · 物理拓扑(真相)

        pulse 连接 A                      pulse 连接 B
   ┌───────────────────┐            ┌───────────────────┐
   │                   │            │                   │
 ┌─┴─┐   WS (会断)   ┌─┴──────────┴─┐   WS (会断)    ┌──┴──┐
 │arm│◄─────────────►│    head      │◄─────────────►│tent │
 └───┘               │ (总在线)      │               └─────┘
 频繁离线             │ pulse 端点×2  │               "该"一直在线
 (锁屏/切后台)         │ + 桥接 + store│                (笔记本)
                     └──────────────┘

 • WS 的物理两端 = (arm, head) 和 (tentacle, head)。arm↔tentacle 无直连。
 • head 是唯一同时握着两条 WS、且总在线的节点。
 • 因此:能替"随时离线的端"扛消息的,只有 head。
关键不对称 发送方(arm 手机)比接收方(tentacle 笔记本)更容易离线。 要在"发送方先离线"和"接收方先离线"之间架桥,桥必须落在那个不离线的节点 —— 只有 head。 这就是 durable store 必须在 head、不能在端点的根本原因。

4 · 三层模型 & E2E 正交

我之前反复把 E2E 拉进 pulse 的设计考量,是层次没分清。正确的心智模型是三层,各管各的,互不知道对方:

kraki 应用层
知道 session / approve / delete。在 payload 交给 pulse 之前,已经 E2E 加密好了。决定"这条 delete 值得 durable"。
knows: 内容、加密、消息类型、哪条该 durable
pulse 传输层
只认 bytes + seq + ack + durable 位。逐跳可靠。运的 payload 恰好是密文,但它不知道那是密文,也不知道是 delete_session。
knows: 一条会断的 link、序号、确认、durable 能力位  |  不 knows: kraki、加密
WS 物理层
arm↔head、tentacle↔head。运字节,会断。
knows: 字节流的生死
E2E 边界不用动 kraki 该加密的照旧加密。pulse 拿到的就是一坨它不理解的 bytes。head 作为 pulse 端点读 pulse 传输头(seq/ack/durable), 但 payload 里 kraki 加密的内容 head 依然读不了。head 多看见的只是传输元数据(它今天本来就看得见 relaySeq)——没有 E2E 回退。

5 · 逐跳 pulse & head 桥接

head 跑两个 pulse Endpoint(每条 WS 一个),中间是一个 store-and-forward 桥:

  arm ──payload(密文)──►  [ Endpoint_A ] ──deliver──►  head 桥接逻辑
                          (arm↔head hop)                    │
                                                            │ 看 `to` = tentacle
                                                            │ tentacle 在线?
                                              ┌─────────────┴─────────────┐
                                        在线 │                            │ 离线 + durable
                                             ▼                            ▼
                                    [ Endpoint_B ].send          durable store (落盘)
                                    (head↔tentacle hop)          等 tentacle 重连
                                             │                            │
                                             ▼                   tentacle 重连 → 从 store
                                          tentacle                取出 → Endpoint_B.send
  • 每跳独立 seq 空间:arm↔head 有自己的 1,2,3…;head↔tentacle 另有 1,2,3…。head 把 hop A 收到的 payload,当作新消息在 hop B 发出。
  • 端到端可靠 = 两段逐跳可靠 ∘ head 桥接正确。head 的桥(含 durable store)在交接时不丢不重,是新的正确性要求。
  • 这恰好收编了 head 今天的 relaySeq + pending_messages —— 它们本就是这套 store-and-forward 桥的手写前身,只是没有统一语义。
  • head 从不解密:两跳的 pulse payload 都是同一坨 kraki 密文;head 收下、存、转发,never 读内容。

6 · durable 能力模型

这是整套设计的核心,也是它不破哲学的地方。durable 是端点在连接建立时协商出来的一个能力,不是消息的应用语义,也不是 link 的特殊拓扑。

握手时:端点声明能力

// pulse HELLO 握手里,每个端点声明:
{ durable: { supported: bool, maxRetentionMs: number } }

//   arm      端点 →  { supported: false }
//   tentacle 端点 →  { supported: false }
//   head     端点 →  { supported: true, maxRetentionMs: 30 * 86400_000 }  // 30 天

发送时:看对端能力决定能不能打 flag

endpoint.send(payloadBytes, { durable: true })

// pulse 内部:对端(peer)supported === true 吗?
//   是 → 这条 frame 带上 durable 位(head 可读的传输头,非加密)
//   否 → durable 无意义(降级为普通,或由集成层判定为 fail-fast)
为什么不破哲学 pulse 核不认识 delete_session,也不认识 relay 是 kraki 的 head。它只知道两件纯传输的事: ①"对端声明了 durable=true"(一个能力位);②"这条消息应用要 durable"(一个 flag)。 "delete_session 该 durable" 是 kraki 集成层的决定;"head 是那个能存的端" 是 部署的事实。 pulse 接口上出现的每个词(supported / durable / retention)都是传输概念,没有一个暗示"上层有特殊消息"。
durable 位为什么必须 head 可读(而 payload 不可读) head 解不开 kraki 密文,所以它无法从内容判断"这条是 delete_session(该存)还是 approve(该 fail)"。 于是发送方(arm,它读得懂自己的消息)必须通过一个 head 可读的传输位告诉 head:"这条请你存,别 fail"。 这个位是 pulse 传输头的一部分,不涉及内容 —— E2E 完好。

7 · 走查:delete_session,tentacle 离线

1
arm:用户点删除。不 block —— 本地乐观移除该 session,UI 立即更新。
2
arm 的 kraki 层:把 delete_session 的 payload 加密(E2E,for tentacle 的 key),标记"这条 durable"。
3
arm 的 pulse(arm↔head hop):send(密文, {durable:true})。对端 head 声明 supported=true → frame 带 durable 位。进 arm 的 outbox。
4
head 的 Endpoint_A:可靠收下,deliver 给 head 桥接逻辑,并回 ack 给 arm。
5
arm:收到 acked → 这条已安全抵达常在线的 head。arm 现在可以立刻离线,delete 不会丢。
6
head 桥接:看 to = tentacle,tentacle 离线;因为这条带 durable 位 → 存进 durable store(落盘,存的是读不懂的密文 blob)。
7
tentacle 重连 → head↔tentacle 的 Endpoint_B 建立 → head 从 store 取出这条 → Endpoint_B.send → tentacle 的 pulse deliver → kraki 解密 → 执行删除 → ack 回 head → head 从 store 清掉。
解决了什么 arm 只需"活到把消息交给 head"这一瞬间,不需要活到 tentacle 上线。这正是端点自持久解决不了、而 head-store 解决的问题。

8 · 走查:approve,tentacle 离线(fail-fast 对照)

1
arm:用户点 approve。乐观切绿(但记住旧状态以便回滚)。
2
arm pulse:send(密文),不带 durable(kraki 判定 approve 不该 durable)。
3
head 桥接:to = tentacle 离线;这条没有 durable 位 → head 不存 → 回一个失败信号给 arm(ref 通道)。
4
arm:收到失败(或本地超时没等到 acked)→ 回滚旧状态 + 显示 error → 用户可重试。
为什么 approve 绝不能 durable(审计发现) relay-client 在调 adapter 之前就无条件向所有 arm 广播了 permission_resolved: approved。而 tentacle 重启后那个 permission 的 resolve 回调随进程死了。 所以一条离线缓存 30 分钟后送达的 approve → adapter 静默 no-op,但每个 arm 都被谎报"已批准"。这不是"延迟但正确",是主动骗 UI。 fail-fast 在这里是安全要求,不是偏好。

9 · 消息分类表(审计逐条判定)

哪些消息该 durable(值得 head 替离线 tentacle 存),哪些该 fail-fast,哪些本就不该发给离线设备。

consumer 消息判定理由
delete_sessionDURABLE幂等、无时效;30 分钟后删一样对;无其他 re-sync 路径(删了没有"列表同步"能表达"这个 session 不该存在")。
set_session_mode弱 YES幂等持久配置,迟到应用仍对。但有救济:值会随 session_list 重连同步。建议先不列入,靠 session_list 兜。
set_session_model弱 YES同上,已持久化 + 即时 ack;同样可先靠 session_list 兜。
approve / deny / always_allowFAIL-FAST响应一个已随离线消失的 permission;no-op + 谎报 resolved(见上)。
answerFAIL-FASTquestion 的阻塞 turn 已没;no-op + 谎报 resolved。
send_inputFAIL-FAST延迟送达会造幻影 user_message + 触发用户没要的 turn;聊天惯例=失败重试。
kill_session / abort_sessionFAIL-FAST要 kill/abort 的 turn 30 分钟后早已结束,无物可作用。
create_session / fork_sessionFAIL-FAST延迟会造幻影 session;自带 requestId 正是为当场成功/失败反馈。
mark_readN/Abest-effort 已读游标,过期值意义≈0。
request_session_messages(_range)N/A离线 tentacle 回不了历史,arm 应走本地缓存。
最小 durable 集 建议 = 只 delete_session。mode/model 先靠 session_list 重连同步兜底,以后需要再加。durable 集越小,head store 越简单。

10 · pulse 接口草图(仍是纯黑盒)

// ── 端点构造 / 握手 ──
new Endpoint({
  epoch,
  durable: { supported: false },        // arm / tentacle 这样
  // durable: { supported: true, maxRetentionMs: 30*86400_000 },  // head 这样
})

// ── 发送(应用 → 核) ──
endpoint.send(payload: Uint8Array, opts?: { durable?: boolean }): { seq, effects }
//   payload 永远是 opaque bytes。durable 仅在对端 supported 时有意义。

// ── effects(核 → 适配器)——已有的 + 新增 ──
| { t:'transmit', bytes }
| { t:'deliver', seq, payload }
| { t:'open' } | { t:'close' }
| { t:'reset-inbound', fromSeq, peerEpoch }
| { t:'acked', seqUpTo }                 // (已加) 送达确认 → 乐观 UI 落定/回滚
| { t:'store', seq, payload }            // (新) durable 端点:请把这条持久保管
| { t:'unstore', seqUpTo }               // (新) 已被下游取走,可清

// ── 输入(适配器 → 核)——不变 ──
onConnected / onDisconnected / onBytes / onTick

store/unstore 是给 durable-supported 端点(head)的 effect:核判定"这条需要持久",让适配器落盘; 但"存到哪、按什么 key、何时取"仍是 head 应用层的路由知识(它知道 to=tentacle)。 这是需要你拍的边界之一(见下)。

11 · 对现有代码意味着什么

部分状态说明
pulse 核(Endpoint / wire / acked / dup-ACK)大体可复用Endpoint 是"一条连接"的通用可靠状态机,arm↔head 和 head↔tentacle 同样适用。需加:HELLO 的 durable 能力协商 + frame 的 durable 位 + store/unstore effect。
TS + Swift 双实现 / 66+37 测试 / property保留核的行为契约不变,新增能力增量加测试。
tentacle/arm 集成(把 pulse frame 塞进 E2E 密文)推倒重做那是"端到端 + 哑 head"的错误方向。改成:pulse 是逐跳的传输层,frame 在 head 可读的信封层,包着仍加密的 payload。
head(哑管道)升级为 pulse 端点head 跑两个 Endpoint + store-and-forward 桥。它今天的 relaySeq + pending_messages 被这套统一收编。
head pending_messages(#81)概念保留它不是要删的技术债,是 durable store 的手写前身。语义并入 pulse 的 durable 能力。
诚实说明 之前 Phase 8 的集成(commit "wire pulse into tentacle + arm")基本要重做 —— 但那不是白费:它验证了核是对的、E2E 能干净搭载、以及"发送前不落盘"那个洞的存在。地基挪对位置后,核几乎原样复用,变的是集成接线 + head 升级。
